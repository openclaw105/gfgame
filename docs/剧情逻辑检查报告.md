## 主要问题

### 1. `reject_demo_count` 写入了剧情，但运行时没有应用

剧情源中第6章存在选项效果：

- `scripts/story_content.py:781`：`{"reject_demo_count": 1}`

运行时默认状态也定义了该字段：

- `js/engine.js:63`：`reject_demo_count: 0`

但 `applyEffects` 只处理了好感、信任、舆论、艺术值、误会值等字段，没有处理 `reject_demo_count`。

影响：玩家多次拒绝张艺兴 DEMO 的分支倾向不会累计，`zyxDanceEggCheck` 中 `reject_demo_count >= 3` 的概率惩罚逻辑无法按设计触发。

建议：在 `applyEffects` 中补充 `num("reject_demo_count", eff.reject_demo_count || 0, 0, 99)`。

### 2. 第10章“机缘卡还剩”选项会绕过机缘卡数量校验

剧情源：

- `scripts/story_content.py:1219`：第10章问题“机缘卡还剩，你？”
- `scripts/story_content.py:1221`：用给肖战，`{"chance_card": -1, ...}`
- `scripts/story_content.py:1222`：用给宋威龙，`{"chance_card": -1, ...}`

运行时 UI 面板使用机缘卡时有数量校验：

- `js/engine.js:1054`：`useChanceCard(target)`
- `js/engine.js:1056`：`if (s.chance_card <= 0) return`

但剧情选项走的是：

- `js/engine.js:769`：点击选项直接 `applyEffects(...)`
- `js/engine.js:248`：`s.chance_card += eff.chance_card`

这里没有防止机缘卡变成负数。

影响：即使玩家没有机缘卡，也可以在第10章选“用给肖战/宋威龙”，并让 `chance_card` 变为 -1，同时获得好感奖励。

建议：剧情选项中不要直接写 `chance_card: -1`，改为专门的条件选项；或在 `applyEffects` 中对 `chance_card` 做 `Math.max(0, ...)`，并在选项渲染阶段禁用条件不满足的选项。

### 3. 多个结局按当前剧情数值不可达或高度依赖 UI 机缘卡

按所有剧情选项可获得的最大/最小值估算：

| 字段 | 剧情内范围 |
|---|---:|
| money | 0-133 |
| chance_card | -1-0 |
| xz_love | 最高 100 |
| xz_trust | 最高 100 |
| swl_love | 最高 40 |
| zyx_love | 最高 18 |
| lyr_love | 最高 18 |
| vv_friend | 40-100 |
| vv_gj | 最高 42 |
| public_opinion | 最高 66 |
| art_value | 最高 100 |
| misunderstanding | 最高 41 |

因此：

- `雨停之后，无人归来` 要求 `misunderstanding >= 80 && vv_friend <= -20`，当前剧情范围完全达不到。
- `花开两岸，人间重逢` 要求 `vv_gj >= 70`，剧情最高约 42，需要额外靠机缘卡或补剧情数值。
- `风从片场吹来` 要求 `swl_love >= 75`，剧情最高约 40，需要额外靠机缘卡。
- `听见月光在起舞` 要求 `zyx_love >= 65 && zyx_dance_seen`，剧情最高约 18，需要大量机缘卡且依赖随机彩蛋。
- `灯火两端，旧梦不言` 的显式条件要求 `public_opinion >= 85 && art_value < 55`，当前舆论最高约 66；不过该结局仍可能作为 fallback 出现。

影响：玩家会感觉部分“写在结局表里”的结局实际无法通过剧情选择自然走到。

建议：为每个结局做一条可验证的数值路线，或下调条件；特别是负向 BE 需要增加误会值和降低薇薇友情的分支。

### 4. 分支中的 `beat.effects` 会在同一节点重复渲染时重复应用

运行时：

- `js/engine.js:864`：只要渲染到含 `effects` 的剧情节点，就会 `this.applyEffects(beat.effects)`

分支队列推进时也会保存进度：

- `js/engine.js:885-916`：分支队列逐条渲染并保存

影响：如果玩家在分支效果节点处刷新页面、回退到该节点或重复触发渲染，金钱/好感等效果可能重复叠加。

建议：给每个已应用效果的 beat id 做一次性记录，例如 `state.applied_effect_beats`，渲染前判断是否已经应用。

## 次要问题

### 1. 角色表情引用存在大量兜底

剧情中大量引用类似 `char_葵_一般`、`char_薇薇_一般` 的 key，素材 registry 会尝试 fallback 到开心/一般/害羞等可用表情。

影响：不一定报错，但表情精确度会下降，尤其是已经强调“尽量用上人物表情素材”的目标。

建议：后续按素材名录补齐角色表情 key，或在生成脚本中把“情绪标签”限定为素材 manifest 已存在的标签。

### 2. 每章“6次选择”的文档表述容易与遭遇选项混淆

第2、4、7章存在额外遭遇选项，这是设计上合理的支线扩展。

建议：文档中改成“每章固定 6 个主线选择，允许插入遭遇/支线选择”。

## 总结

剧情结构层面基本完整：10章、每章6个主选项、分支插入、3/6/9章收束与第10章结局节点都已生成。

当前最需要修的是三类逻辑：

1. 让所有剧情字段都能被 `applyEffects` 正确应用。
2. 给机缘卡等资源型选项加条件校验，避免负数和白嫖分支。
3. 校准结局条件与剧情数值路线，保证每个结局至少有一条自然可达路径。
